# 需求取捨決策

> 這份文件目前是 Codex 依 Release 01 訪談模擬產生的草稿。最後決策必須由人類刪改、補充或重新勾選。

## 我決定 v1 優先服務的使用者

請勾選一個主要對象，也可以補充原因。

- [ ] 回報者
- [x] 資訊整理者
- [ ] 行動者

原因：目前三個 persona 的共同風險都指向「未確認資訊被誤認為可行動」。v1 先服務資訊整理者，可以優先把原文、人工判斷、查核狀態和不能直接變成任務的原因分清楚。

## 這個使用者最想完成什麼

- 把混亂的 Phase 0 原始資訊整理到下一位協作者能理解。
- 保留原文和人工判斷的差異，不把整理狀態包裝成查證結果。
- 找出哪些資訊需要人工確認、不能直接相信、不能直接變成任務。

## 這個使用者最怕什麼錯誤

- 模糊資料被直接變成任務。
- AI 或整理者補了原文沒有的資訊，下一位協作者卻以為那是事實。
- 「已人工審核」被看成「已確認」。

## 我今天暫時不處理的需求

- 暫時不把 v1 做成正式派工或行動決策工具。
- 暫時不新增後端、資料庫、外部 API、地圖或真實資料查核。
- 暫時不讓 AI 自動產生可派發任務。
- 暫時不處理真實照片、檔案上傳或附件保存。

## 我不採用的 AI 建議

- AI 建議：把三個 persona 的所有需求都整理成 v1 待辦清單。
  - 不採用原因：這會讓 v1 scope 太大，也會把需求分析誤當成已決定的功能。
- AI 建議：把上傳者的資料視為比較接近事實。
  - 不採用原因：上傳者可能是本人、家屬、現場志工或轉述者；來源方式不等於查核結果。

## 必須保留人工確認的判斷

- 是否能把一筆資訊視為已確認。
- 是否能把一筆資訊轉成志工任務。
- 是否存在地點、時間、來源、身分或安全風險的關鍵缺漏。
- AI 或整理者推論是否超出原文。
- 上傳資料即使送出後可被查詢，也仍然不能自動視為已確認。

## 今天明確不做什麼

- 不修改 `src/`；Release 01 是需求分析，不是實作階段。
- 不把未確認內容顯示成已確認。
- 不查外部社群、地圖、政府資料或新聞來補內容。
- 不放入真實個資、真實災情資料、真實地址或真實電話。

## 我仍不確定的地方

- 小隊是否同意 v1 優先服務資訊整理者，而不是回報者或行動者。
- 「已人工審核」是否應改名，或只需要更清楚的輔助說明。
- 災民上傳入口在 v1 是否保留，或先移出主要流程。
